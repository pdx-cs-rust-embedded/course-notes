Embedded Rust Course Notes

Embedded Rust Course Notes

This course is intended to teach you "Embedded Rust". To start, let's keep this simple:

  • The Rust Programming Language is a modern language for safe, performant and friendly low-level programming — really nice for our needs.

  • Embedded Systems are systems that typically:

    • Are focused on direct interactions with electronic hardware.

    • Are too small and slow to run a "real operating system" comfortably.

    We will use the BBC MicroBit v2 (MB2) as the target embedded system for this course.

MB2 Documentation

  • The official MB2 Hardware Documentation is really excellent.

  • We will frequently refer to the NRF52833 manual.

  • This page from Mike Spivey's Bare Metal Micro::Bit book (which I can't find anyplace) contains excellent programming-language-independent information about the board.

Important Readings

  • These notes heavily rely on the Rust Embedded Discovery Book ("Disco Book") for the MB2. You should read that along with this course: we will reference this book quite a bit. That said, the MB2 Disco Book is somewhat incomplete and a bit dated. Work is in progress to correct these deficiencies.

  • We highly recommend and occasionally reference the Bare Metal Rust addendum to the excellent three-day Comprehensive Rust course.

  • The Tweede Golf Embedded Workshop book is full of good things. Parts of it will be referenced here.

What You Need

You'll need some stuff and some knowledge for this course. Here's the list.

Hardware

For this course you will need to have an MB2. You can currently purchase a bare board for about US$15 from a number of sources.

You will also need a kit of hardware that we are going to use for some of the experimentation in this course. This consists of an adapter from the MB2 to a breadboard, a breadboard and some wires, and a bunch of little cheap devices to plug into the breadboard and wire up.

The current parts list is maintained on Google Sheets. Note that the current kit price of around US$30 is based on cheap bulk purchases. If you try to buy these things in small quantities it will likely cost quite a bit more. There is nothing particularly critical about most of these: a knowledgeable person should be able to substitute part numbers pretty freely.

A Computer

A laptop or desktop is needed to program the MB2.

  • Linux is definitely easy mode for this purpose. The examples and the course will use Linux when a "host" OS is needed.

  • If working on a Windows box, exploring Windows Subsystem for Linux 2 (WSL2), which comes with your modern computer, is a reasonable plan. That said, most things can be done directly on Windows.

  • A Mac should also work, but the authors don't have one or use one. (At least in modern times, Bart used to program the 128K Mac using the Apple Lisa.) You're more on your own with a Mac.

Rust

We expect you to have gotten some Rust knowledge and experience before starting here. If you don't have any Rust please work quickly through the first six chapters of The Rust Programming Language.

Rust is syntactically quite similar to other C-like languages, and semantically similar to C++ or Java but more capable than either. We will be there to help you if you get stuck, so don't be discouraged: you can do this.

There's going to be some fairly "advanced" Rust in this course. Some low-level stuff, some slightly fancy Rust types, and even a little bit of async-await. We will review the needed material, especially the material beyond the basics, as we go.

Please make sure you have a working Rust installed on your host machine before taking this course. Rustup will get you going.

Basic Computer Systems Knowledge

You should be aware that CPUs have registers and ROM memory and RAM memory, part of which is used as a stack. You should be aware that CPUs execute machine instructions. You should have some idea of what these things are/do. Any University introductory systems course should be more than adequate. We will refresh you on these things as we go.

What You Won't Need

There are some things that you might think you will need that you do not: we will provide them.

  • Electrical / electronics knowledge. We will get you what little you need early. If you've never seen an electricity before, this course is still very much for you.

  • Detailed knowledge about the inner workings of embedded systems. That's a big part of this course.

About Embedded Systems

Describing exactly what an "embedded system" is a bit tricky, and sometimes even a source of controversy. Here are some things to think about.

System Size and Speed

Embedded systems come in several ranges of capability: from small to smaller to smallest.

  • The biggest and fastest embedded systems might be able to run a fancy operating system like Linux. Even then, it will be a really cut-down version. Most run ARM, RISC V, or some other "special" instruction set.

  • The next tier is machines like the MB2 we will be using. We will discuss this machine later, but to summarize it's an ARM 32-bit 64MHz 128KB RAM 256KB flash machine. As such, it will run either a very simple cut-down "embedded OS" or just "bare metal". The MB2 is a "System on Chip" (SoC), meaning that all the core functionality including many peripheral drivers are on a single chip.

  • The final tier is really tiny machines, with specs like 32KB ROM 16KB flash. These may be SoCs, or may just be bare tiny CPUs. These include 16-bit and 8-bit machines. They are stupidly cheap and great fun. They are almost always run bare metal.

Interfacing: Pins

Another big distinguisher of small and embedded systems is that they usually can make use of wires connected directly to the CPU pins for interfacing. These "pins" allow direct electrical connection of signals to the computer, without intervening layers such as USB.

Pins are fantastic for HW cost saving and for doing really custom things. They are also a giant pain. We will rue and eventually rule them in this course.

Documentation

The docs for small and embedded systems are special. Really special. They are written in a shorthand that is somewhat comprehensible to electronics folks and somewhat comprehensible to low-level programmers.

SoC docs are usually big. The MB2 has an NRF52833 processor: the NRF52833 manual is over 500 pages. For a "small" system, there is a lot of functionality here: the NRF manual is dense in spite of its size.

SoC and hardware docs are written in a language of abbreviations. For example, the NRF52833 "low-power comparator" is consistently referred to as LPCOMP. The doc leads with "0-VDD input range", "eight input options (AIN0 to AIN7)" and goes from there. You will be taught a bit about how to read this sort of thing in this course.

Getting Started

MicroBit 2 Setup

We are working with the BBC Micro::Bit v2 (MB2). It is a fairly high-end embedded system.

The MB2 has 512KB flash, 128KB RAM, and runs at 64MHz (64 million instructions per second). It can run MicroPython, although we will never speak of it here. The peripheral set includes some neat stuff. We will poke at many of the peripherals.

The MB2's processor is the NRF52833. This is a 32-bit part that runs the ARM instruction set. ARM machine instructions are quite different from the x86 instructions your host machine probably uses. The 32-bit ARM Instruction Set Architecture (ISA) is called Cortex. The ISA has subsets for smaller chips: M0, M3, M4, M7. The NRF52833 is a Cortex M4 part.

Setting Up The Rust Toolchain

The MB2 doesn't have an OS. You can't just run code compiled for your host on the MB2 because your host isn't a 32-bit ARM (probably). You can't compile for the MB2 on the MB2: it's too tiny and slow to run the compiler.

The solution is "cross-compilation": compile programs to ARM-32 on your host machine and then move them over. For many languages, cross-compilation isn't even possible. For C/C++ it is, but it is a serious pain.

For Rust, cross-compilation is a breeze. The standard tools support it. You tell Cargo that you want to build an ARM-32 version of your code, and it just… does. You move it over and run it.

You describe what you want the Rust toolchain to work with by giving it a "target", either as a command-line argument or in configuration. You can say rustup target list to see a list of all the targets available from Rust: currently about 90 targets coering about 35 ISAs.

The MB2 wants the thumbv7em-none-eabihf target.

  • thumbv7em tells the toolchain to produce/use ARM instructions suitable for (confusingly) both Cortex-M4 and Cortex-M7 processors. The Thumb instruction set is a compacted form of the ISA for use when memory-limited: it trades more decode logic for smaller program size.

  • none means that there is no host operating system or libraries on the target machine.

  • eabihf is the Extended ABI (Application Binary Interface) with Hardware Floating-point. Yes, we have an FPU on this chip.

Cargo Config For Embedded

Rust is special in that it is really well set up for cross-compilation to embedded targets.

The probe-rs project is set up to enhance these capabilities further. Ferrous Systems is a thing in this space.

  • Let's take a look at a sample Rust program and talk about its Cargo config a little: https://github.com/pdx-cs-rust-embedded/blinky-rs

Linkers and other nightmares

  • One thing the Rust tooling hides from us in part is linking/loading. Once the compiler is done, all that program code, initialized data, etc has to be set up in a form that the MB2 bootloader can take.

  • Our tooling invokes flip-link, which is just a script that invokes the host platform's ARM linker, or rust-lld which is a link to the LLVM linker. These days most linkers will work with many ABIs including ARM.

  • Everything expects ELF now. ELF is a format for object and executable files. The program loading tool will read info from the ARM executable and set the bootloader up to put things in the right place.

  • "The right place" is pretty much per-board. Rust uses a file link.x to figure that out: it lives in the target directory until needed. Let's take a look…

Bootloading and debugging

  • There's a little bit of bootloading in ROM on a separate nRF52820. Essentially, there's some support for on-chip debugging, and this includes something that can take a binary by USB and put things where it's told.

  • The specific protocol here is CMSIS-DAP. Mostly don't ask.

  • For Rust, the probe-run command should be installed, and sets this all up. The cargo-embed utility is also involved often.

  • A more generic solution is provided by OpenOCD. For other boards, and other languages, this is an invaluable tool. We likely won't use it here.
